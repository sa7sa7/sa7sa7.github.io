Complex event processing: principes, outils et applications.

Quel potentiel pour ce concept ?

Posted by SALAH on February 16, 2018

Introduction

Les évènements sont partout : dans nos applications, voitures, appareils, serveurs, et même dans nos cerveaux.
De nos jours, on a de plus en plus d’objets connectés.
Le défis du jour est d’analyser cette quantité énorme d’évènements : un problème fortement lié au Big Data.

Cas d'utilisation

Définition du concept

Selon la définition fournie par EPTS (Event Process Technical Society) dans son glossaire publié fin 2008, Complexe Event Processing, (en français, traitement des événements complexes), est tout calcul/traitement exécutant des opérations sur des événements

Deux dérivés conceptuels se distinguent:

  • ESP (Event Stream Processing): traite des streams d’événements ordonnés dans le temps
  • CEP (Complex Event Processing): réfère à un processus de traitement d’événements qui assume avoir un Event Cloud comme input, par conséquent, il n’y a aucune supposition quant à l’ordre d’arrivée des messages.

On note la différence entre deux termes interéssants:

  • Complex event (un événement complexe): un événement traduisant une abstraction d’autres événements. Exemple: une transaction en ligne est une abstraction de plusieurs événements (dépôt de l’article dans le panier, saisie d’un code réduction, saisie du n° de la carte bancaire, etc.)
  • Composite event (un événement composé): un événement construit par combinaison de plusieurs événements utilisant des constructeurs d’événements particuliers, comme une conjonction, disjonction, etc.

La plupart des solutions CEP et concepts peuvent être classés en deux catégories principales :

  • Calcul orienté CEP: Axée sur l'exécution d'algorithmes en ligne en réponse à des événements entrant dans le système. Un exemple simple consiste à calculer en permanence une moyenne basée dans les données contenues dans les événements entrants.
  • détection orientée CEP: Axée sur la détection des combinaisons de modèles d'événements appelés situations. Un exemple simple consiste à détecter une situation par la reconnaissance d'une séquence spécifique d'événements.

C'est quoi un évènement? (déjà!!)

Il existe trois différents sens du mot "Event" qui sont utilisés:

  • Event: signifie tout ce qui se passe, ou un changement d'état. Il existe des différences intéressantes entre les événements et les changements d'état.un événement peut être réel ou imaginaire, passé ou futur. Les films et les rêves ont des événements imaginaires.
    On note une différence entre communiquer des événements futurs(prédire un défaut dans une machine) et analyser des évènements passé(des températures pour un système, historique d'opérations...)
  • Event: Abréviation d'un "Event Object". En informatique, un objet événement est généralement une donnée - une chaîne de bits, peut-être sous la forme d'un message JSON ou XML. Le contenu d'un objet événement est déterminé par la personne qui a conçu le système. Il contient souvent un "time stamp" pour indiquer quand est ce que l'évènement a été généré, reçu, lu...
  • Event: signifie une "notification", un "signal" qui résulte d'un évènement qui se produit. C'est la communication d'un composant (émetteur ou producteur) vers un autre composant (récepteur ou consommateur).
    En programmation, une notification est mise en œuvre en tant que transfert de données, transfert de contrôle ou les deux.

Un exemple en traitant l'évènement en tant que "Event Object"

Ce qui est connu: Stocker des donnée dans des tables, Concevoir un base de donnée relationnelle. Les colonnes représentent des propriétés de l'entité. Les lignes représentent une instance ou un exemple.
On peut capturer la situation courante de l'application: on met à jour les attributs. De plus, les données sont normalisés : pas de duplication.
Problème: Faut faire des jointures, requêter beaucoups de tables pour déduire des conclusions. Ci-dessus un exemple d'un évènement de "Death" dans un jeu.
Les avantages d'une telle représentation:
  • Une structure riche
  • Data pas normalizée
  • Pas de chemas précis pour la Data

Un exemple en traitant l'évènement en tant que "Event notification", "Signal"

Cet exemple suppose qu'on a déjà notre flux d'évènements ordonné dans le temps. Pour l'obtenir, on utilise un des Frameworks qui existent: Kafka, SparkStream...

Illustration avec Apache Kafka

  • Kafka est un système de messagerie de type publication/abonnement conçu pour remplacer les courtiers de message traditionnels.
    Initialement créé et développé par LinkedIn, puis passé en open source en 2011, Kafka est actuellement développé par la Apache Software Foundation. Il vise à exploiter les nouvelles infrastructures de données rendues possibles par les clusters massivement parallèles.

Un exemple avec une architecture Cloud by Google

  • ClickStreams, UserActions: Interactions avec des applications dans Cloud ou On-promises:
    • Ajouter des articles à un panier ou à une liste de souhaits
    • Les transactions financières
  • Google Cloud Pub / Sub: couche de messagerie, service de transport de messages.
  • Google Cloud Dataflow: transforme, enrichit, agrège ou applique des données générales à partir de sources hétérogènes multiples et les formate en une seule représentation cohérente.
  • Stockage d'événements: dans le service de base de données NoSQL
  • Exécution des règles
  • Actions sortantes: notifications pour les appareils mobiles, notifications pour d'autres applications ou envoi d'e-mails aux utilisateurs.
  • Stocker les résultats de l'exécution des règles

Large Domaine d'applications

  • Traçabilité par le biais de puces RFID: Fiabilisation de la traçabilité d’objets dans une chaîne logistique par exemple.
  • Business Activity Monitoring (BAM)
